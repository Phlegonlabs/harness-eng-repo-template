# Orchestrator Workflow

This repository uses a single orchestration model for Codex and Claude.
The runtime now treats long-running work as a contract-driven loop, not a single "pick next task and hope" step.

The planning and execution sequence is:

1. `bun run harness:init -- <name>` personalizes the engineer-template baseline and leaves the repo ready to customize.
2. `docs/product.md` captures product intent, success criteria, scope, and proposed milestones.
3. `docs/architecture.md` captures system shape, constraints, module boundaries, and milestone-splitting constraints.
4. Optional: `bun run harness:context:sync` normalizes external product, architecture, and design inputs into canonical repository surfaces.
5. `bun run harness:plan` validates both docs and synchronizes:
   - `docs/progress.md`
   - `.harness/state.json`
6. Milestones become the top-level executable units.
7. `bun run harness:orchestrate` prepares the next task by creating a task contract and initial handoff artifact.
8. The implementer executes the task inside the contract scope.
9. `bun run harness:evaluate --task <id> --all` runs the independent evaluator and records pass/fail plus a fresh handoff artifact.
10. Different milestones may run in parallel through isolated git worktrees.

Optional path:

- `bun run harness:discover --reset` re-enters guided discovery mode and rewrites the PRD/architecture from interview answers.

## Planning Contract

Backlog generation is blocked until all three conditions are true:

- `docs/product.md` no longer contains template placeholders
- `docs/architecture.md` no longer contains template placeholders
- `docs/product.md` includes a `## Proposed Milestones` section with milestone bullets

Those conditions can be satisfied in two ways:

- `harness:init -- <name>` seeds a ready engineer baseline with valid milestones
- `harness:discover --reset` walks the team through guided PRD and architecture questions
When the conditions are met, `harness:plan` generates milestone records and placeholder tasks.

## Milestone and Task Model

Milestones are the unit of parallelism.

Each milestone records:

- `id`
- `title`
- `goal`
- `status`
- `dependsOn`
- `parallelEligible`
- `affectedAreas`
- `worktreeName`

Each task records:

- `id`
- `milestoneId`
- `title`
- `kind`
- `status`
- `iteration`
- `contractStatus`
- `evaluatorStatus`
- `dependsOn`
- `affectedFilesOrAreas`
- `requiredSkills`
- `validationChecks`
- `evaluationGates`
- `acceptanceCriteria`
- `contextRefs`
- `artifacts.contractPath`
- `artifacts.latestEvaluationPath`
- `artifacts.latestHandoffPath`

Tasks inside one milestone are serial by default.
Only read-only sidecars may overlap inside the same milestone.

### Task Lifecycle

Task status is now explicit:

```text
pending -> contract_pending -> contract_approved -> in_progress -> evaluation_pending -> done
                                                             \-> blocked
```

Rules:

- A task is not done when code "looks ready"; it is done when evaluator output passes.
- Evaluator gates run sequentially and later gates are skipped after a blocking failure.
- Evaluator commands may retry on infrastructure-style failures such as timeouts or process launch issues, but gate ordering and blocking semantics do not change.
- `bun run harness:evaluate --task <id> --gate <gate-id>` previews one gate without advancing task lifecycle.
- Every execution round writes a handoff artifact.
- Failed evaluation increments task iteration and either returns the task to `in_progress` or blocks it when retry/stall limits are exceeded.

## Parallel Dispatch

`bun run harness:parallel-dispatch -- --apply` may allocate worktrees for milestones only when:

- the milestone has no unmet dependency
- `parallelEligible` is true
- its `affectedAreas` do not overlap with active milestones

The dispatcher records worktree ownership in `.harness/state.json`.

## Skill Loading

Skills follow progressive disclosure:

1. The runtime reads `harness/skills/registry.json`
2. It selects candidate skills from the current phase and task kind
3. It merges those candidates with task-required skills
4. It may add skills through deterministic file-pattern matches against `affectedFilesOrAreas`
5. It applies registry routing policy such as weights and max-loaded-skill caps
6. It records the loaded skills, active guardrails, and active exit criteria in `.harness/state.json`

Frontend tasks may also pick up optional design context through deterministic contract injection based on affected areas and the populated `docs/design/` surfaces.

During evaluation, any skill-linked exit criteria run through the same evaluator command path as task validation checks. Skill exit gates do not create a second task lifecycle; they become normal evaluation blockers when they fail.

Codex and Claude must follow the same registry and triggers.

## Command Loop

Primary execution loop:

```bash
bun run harness:status --json
bun run harness:compact
bun run harness:context:sync --design-system <path>
bun run harness:guardian --mode preflight
bun run harness:orchestrate
# implement work inside the current task contract
bun run harness:evaluate --task <id> --all
bun run harness:self-review --report
bun run harness:docs --report
bun run harness:quality --score
```

The orchestrator owns contract synthesis and resume artifacts.
The evaluator owns task-level pass/fail.
The status command owns the structured summary surface for agents.
The compact command owns the concise repository snapshot for resumes and handoffs, including recent task history and resume sequencing hints.
Guardian commands own repo-level preflight, stop, and drift enforcement.
`bun run harness:validate` remains the default local gate before handoff or push.
`bun run harness:validate:full` remains the CI-equivalent gate when harness runtime regressions must be exercised.
